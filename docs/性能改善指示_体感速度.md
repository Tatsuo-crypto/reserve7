# 体感速度の改善指示（2026-07-08 / Fable5 → Sonnet実装指示）

オーナー報告: 起動時の予約画面・タブ切替で3秒以上、予約ボタン押下後に数秒。コードを実測し、構造的な原因を3つ特定した。**上から順に実施すること（1が最大の原因の可能性が高い）**。

---

## 原因1: サーバーの設置場所が日本になっていない（最重要・即修正）

`vercel.json`にも`next.config.js`にもリージョン指定が**一切ない**。Vercelの関数は未指定だと米国東海岸（iad1）で動く。Supabaseが東京にある場合、**すべてのAPI呼び出しが太平洋を2往復**しており、何をどう最適化しても一律1〜2秒の下駄を履かされる。「何度直しても遅い」の正体はほぼ確実にこれ。

**指示**:
1. Supabaseプロジェクトのリージョンを確認（東京=ap-northeast-1のはず）
2. `vercel.json`に`"regions": ["hnd1"]`（東京）を追加し、関数をSupabaseと同じ場所に置く
3. 修正前後で同一APIのレスポンスタイムを計測して比較を記録（これだけで500ms〜1.5秒/リクエスト縮む見込み）
4. 納品手順書にも「買い手のSupabaseは必ず東京リージョンで作成」を追記

## 原因2: 予約ボタンがGoogleカレンダーの書き込みを待っている

`api/admin/reservations/route.ts` 331行・394行で、予約保存のリクエスト中に`await calendarService.createEvent(...)`を実行している。つまり**ユーザーはDB保存＋Google API応答（数百ms〜数秒）の合計を待たされている**。会員側の予約APIも同様なら同じ問題。

**指示**:
1. APIの応答は**DB保存が完了した時点で返す**。Googleカレンダーへの書き込みは応答後の非同期処理に移す（Vercelなら`waitUntil`が使える）
2. カレンダー書き込みが失敗した場合はDBに同期失敗フラグを記録し、次回cron実行時にリトライ（黙って握りつぶさない）
3. 削除・変更時の`deleteEvent`等も同様に非同期化
4. 効果: 予約ボタンの待ち時間が「DB保存のみ」（原因1修正後なら数百ms）になる

## 原因3: タブ切替のたびに全データを再取得している（SWR未導入）

P章第2段で決定済みのSWR導入が**未着手**（コード内に`useSWR`の使用ゼロ）。タブを行き来するたびにネットワーク往復が発生している。

**指示**:
1. SWRを導入し、`useWeeklyProgress`と各タブのデータ取得をuseSWR化（P-2の決定どおり、ダイエット領域から）
2. 設定は**キャッシュ優先**: 前回のデータを即座に表示→裏で更新（stale-while-revalidate）。タブ切替は体感0秒になる
3. ログイン直後に主要タブのデータを裏で先読み（prefetch）し、初回のタブ移動も待たせない

## 原因4（仕上げ）: 「待っている感」の演出不足

ネイティブアプリが速く感じるのは、実際の速さに加えて**押した瞬間の反応**があるから。

**指示**:
1. ボタンは押した瞬間に押下状態＋処理中表示に変える（0.1秒以内の視覚反応）
2. 予約作成は楽観的更新: API応答を待たずに一覧に予約を表示し、失敗したら戻してエラー表示
3. 読み込みはスピナーではなくスケルトン（商品診断の診断-4と同一作業。二重実装しないこと）

## なぜ「他のアプリ」は速いのか（オーナー向け解説）

ネイティブアプリはデータの多くを端末内に持ち、開いた瞬間は手元のデータを表示してから裏で更新している。今回の修正はWebアプリで同じ体験を作るもの: ①サーバーを近くに置く（原因1）②待たなくていいものを待たない（原因2）③一度見たものは覚えておく（原因3）④押した瞬間に反応する（原因4）。この4つが揃うと、Webでもネイティブに近い体感になる。
