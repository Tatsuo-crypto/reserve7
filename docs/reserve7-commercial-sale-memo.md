## 1. 背景・状況

- reserve7はもともと自社（T&J GYM）用に作った予約・ダイエット管理アプリ。N-5決定（デザインシステム統一メモ）で「デザイン統一は販売パッケージ化の前提条件」と位置づけ、直近のPR群（トークン統一・黒ベース化・アイコン統一・コントラスト調整等）はその準備として実施済み。
- 技術構成: Next.js + Supabase(DB) + NextAuth(管理者認証) + トークン認証(会員・トレーナー) + Google Calendar連携。**ダウンロード配布型のアプリではなく、ホスティングを伴うマルチテナントSaaS**。
- オーナー本人はプログラミング・アプリ運営の初心者であり、契約・法務・インフラ運用の実務経験もこれから。

## 2. オーナーが答えを求めている問い

1. 他社へ販売する際、**絶対にやってはいけないこと**は何か（法的リスク・将来の自分の不利益になりうる点）
2. 「売り切り」の方がよいのか、それとも別の形（サブスク等）が現実的か
3. 自分が初心者であることを踏まえて、**そもそも現実的な話なのか**
4. 上記を踏まえて、やらない方がよいことを具体的に

## 3. Sonnet側で調べた前提情報（参考・未確定）

以下はWeb検索に基づく一般的な情報であり、弁護士による確認を経ていない点をオーナーにも伝達済み。

**技術構成と販売モデルの相性**
reserve7はSupabase等の継続的なインフラコストを伴うホスティング型のため、「売り切り（一回払い・以後無関係）」とは構造的に相性が悪い。実質的な選択肢は次の2つ:
- (a) 月額利用料（サブスク）でこちらがホスティングし続ける
- (b) ソースコードごと渡し、購入側が自前でインフラを持つ「セルフホスト型の買い切りライセンス」（この場合も設定サポート等の問い合わせは発生しがちで、完全に無関係にはなりにくい）

**最低限のリスク対応として挙げた項目**
- 利用規約・免責条項（正確性・完全性・安全性の無保証、間接損害の免責等）の整備
- 個人情報保護法対応（会員の氏名・体重・食事記録・予約履歴を扱う以上、プライバシーポリシー整備・利用目的の明示・委託先管理の説明責任が必要。2026年は法改正が進行中で中小・個人事業者も対象）
- 体重・食事記録は、医療機関の診療プロセスを経ずジム利用者が自主記録する分には「要配慮個人情報」には該当しないと考えられるが、機密性の高い情報として丁寧な扱いが必要
- 第三者サービス（Google Calendar API、Supabase）の利用規約が「複数の顧客に商用展開すること」を許容しているか、契約前に要確認（アカウント停止リスク）
- サポート範囲・保守期間を契約書で明記しないと「売った後もずっと無償で直してもらえる」という期待値のズレが起きやすい
- 個人事業主のまま複数社と契約する場合の責任集中リスク。事業規模が見えてきた段階で法人化・賠償責任保険を検討

**現実的な進め方として提案した内容**
- 最初から完璧な契約書・法人・保険を揃えようとせず、まず1社（知人のジム等）に月額利用料で試験導入し、運用しながら体制を育てる
- 最低限「利用規約」「プライバシーポリシー」「免責条項」の3点は最初から用意し、雛形＋専門家レビューで揃える

## 5. 追加相談（2026-07-07 / 実際の売り方について）

前回の「絶対にやってはいけないこと」は理解した上で、オーナーから続けて次の相談があった。

1. **実際にどう売っていくのがよいか**（具体的な進め方・営業の仕方）
2. **サブスクにするならいくらくらいが妥当か**
3. **相手（顧客）によって機能を分けて売った方がよいか**（松竹梅・機能ティア分け）
4. **「売り切り」にした場合の具体的なリスク**

### Sonnet側で調べた参考情報（追加分）

**価格帯の相場感**
日本国内のジム向け予約・会員管理SaaSは、規模に応じて概ね次のような相場になっている。
- 小規模（パーソナルジム・1〜2店舗）: 月額1万〜5万円程度
- 中規模以上（複数店舗・オプション込み）: 月額3万〜15万円程度
- オールインワン型（予約・決済・顧客管理込み）で月額1万円台〜からのサービスも存在する
reserve7は現状「予約管理＋ダイエット管理（PFC・体重・週間パネル等）」という2つの柱を持つ点が一般的な予約管理SaaSとの差別化要素になりうる。単純な予約システムより高めの価格帯（例えば月額2万〜4万円程度）を狙う根拠にはなりそうだが、競合の具体的な機能比較はできていないため参考程度。

**機能ティア分け（松竹梅）について**
SaaS一般では「ユーザー数課金」「機能数課金（松竹梅の3段階が定番）」「予約件数課金」などの組み合わせが多い。3段階程度に抑えるのが定石で、プラン数を増やしすぎると顧客が選びにくくなるという注意点がある。reserve7で考えられる切り口としては、例えば「予約管理のみ」「予約＋ダイエット管理」「複数店舗・トレーナー管理込みのフル機能」のような分け方が候補になりうる（あくまで案、確定ではない）。

**「売り切り」の具体的リスク（売り手側）**
- 継続収益がないため、事業として安定しにくい（初期の一括収入は大きいが、その後の収益源がなくなる）
- reserve7はSupabase等の継続的なインフラコストがかかる構造のため、買い切りにしても実質的にホスティングを続ける限りコストだけが継続し、収入は増えないという逆ざやが起きうる
- 「売り切ったら終わり」という設計にしても、実際には顧客からの問い合わせ・軽微な修正依頼が発生しやすく、無償対応が常態化するとサポートコストだけが残る
- 継続的なアップデート・改善の動機が薄れる（サブスクなら解約されないために機能改善を続ける必要があるが、買い切りだとその圧力がない分、競合に劣化していくリスク）

### Fable5に相談したいこと（追加分）

- reserve7の機能構成（予約・ダイエット管理・トレーナー管理・店舗管理等）を踏まえた**具体的な価格帯・プラン設計の方向性**
- **機能ティア分けをするならどの機能を松・竹・梅に振り分けるべきか**（UI/UX上、どの画面をどのプランで出し分けるかの設計にも関わる）
- 実際の売り方（最初の1社の見つけ方、営業トーク、デモの見せ方等）についての助言
- 「売り切り」を選択肢として残すべきか、それとも完全にサブスク一本に絞るべきかの最終判断
