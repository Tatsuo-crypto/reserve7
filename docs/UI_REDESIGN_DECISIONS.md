## A. 会員側の決定

### A-1. ホームのカード構成（最終決定）

ホームは「数字の一覧」から「今日の秘書」に変える。カードは最大4枚、上から:

| # | カード | 表示条件 | 表現 |
|---|---|---|---|
| 1 | 本日のオンラインレッスン | 当日開催時のみ | 開始時刻＋「参加する」ボタン（Meetリンク直結）。1タップで参加できることが目的 |
| 2 | 次回予約 | 常時 | 「7/5(日) 10:00 パーソナル」＋タップで予約タブへ。予約なしなら「予約する」CTA |
| 3 | 体重ひとこと | 記録があるとき | 数値＋先週比のみ。「62.4kg（先週より −0.4kg）」。グラフなし。タップで分析タブの体重グラフへ |
| 4 | 今日の記録 | 常時 | カロリー進捗リング1つ＋「記録する」ボタン。未記録なら記録促し文言のみ |

**隠すもの（すべて分析タブへ）**: P/F/C個別バー、糖質・食物繊維・塩分、水分・歩数・睡眠の個別バー、筋トレ回数バー。

現行の週間目標バー11本は**削除ではなく移設**。分析タブ先頭に「週間目標」セクションとして移し、デフォルト折りたたみにする。見たい人には全部見せる、見たくない人には見せない、が原則。

筋トレは数値バーをやめ、カード4の中に「今週 3/4回 💪」程度の一言バッジで残す。水分・歩数・睡眠はホームから完全撤去（カード4に足すとまた数字だらけに戻るため、初期リリースでは入れない）。

### A-2. 通知タブ → 廃止

独立タブは廃止。ヘッダー右上の歯車アイコン→「設定」画面を新設し、通知許可プロンプトはそこへ移す。加えて、**通知未許可のユーザーにはホーム最下部に許可促しの小バナーを1枚**出す（許可済みなら消える）。タブを畳んでも導線は死なない。

### A-3. ナビ構成 → 5タブ

```
[ 予約 ] [ 記録 ] [ ホーム(中央) ] [ 分析 ] [ プラン ]
```

- **食事タブの入力/分析モード切替を解体**し、「記録」「分析」を独立タブに昇格。現状は分析がタブ内タブに埋もれており、ホームから追い出した数値の受け皿になる分析の格を上げる必要がある
- **オンラインタブは廃止**。当日はホームのカード1が最短導線（現状より1タップ減）。当日以外のレッスン枠一覧は予約タブ内のセクションへ移す。「人と会う予定」は予約タブに集約
- **通知タブは廃止**（A-2）
- プラン縮小構成（input/analyze/progress無効）の場合は「予約・ホーム・プラン」の3タブ維持で現行踏襲

### A-4. 「数字を隠す」際の代替表現のトーン

原則: **ホームでは評価しない、分析タブで初めて数値と向き合う**。

- 一言サマリーは事実＋中立〜前向きのみ。「先週より −0.4kg」は出すが「目標まであと3.2kg」のようなプレッシャー表現はホームでは使わない
- 未達の赤色・警告色をホームで使わない。達成時だけ色（緑/バッジ）が付く非対称設計。未達は無色
- 進捗率の%表示はカロリーリング1箇所のみ。それ以外の数値にパーセントを付けない
- 未記録の日は「サボり」扱いせず「今日の記録はまだです → 記録する」と行動導線に変換

### A-5. 着手順序 → 承認（1点修正）

メモの1→2→3→4を承認。ただし**ステップ1と2は同一PRで実施**する。通知タブ廃止＝ナビ本数の決定であり、ホーム再設計と切り離すと中途半端なナビが一度世に出るため。

1. ホーム再設計＋ナビ5タブ化＋設定画面新設（1PR）
2. 分析タブ整理（週間目標セクション受け入れ、記録/分析タブ分離）
3. 予約タブへのオンラインレッスン枠統合
4. プランタブ微調整（現状維持でも可）

**同時にやること**: HomeTab再設計時に週間実績/目標の計算ロジックを共通フック（例 `useWeeklyProgress`）に抽出し、ProgressTab.tsx（admin/diet-plan用）と一本化する。どうせHomeTabを書き直すので、重複解消はこのタイミングが最安。

## D. 管理側 追加決定（2026-07-04 / v2メモへの回答）

裏取り済みの前提: BlockedTimeModalの外部参照ゼロ、`/admin/calendar`の参照元は`reservations/new`のリダイレクトのみ、を確認。加えて**trainerTokenは`trainer/[token]`系全体で使われるトークン認証の仕組みであり、`/dashboard`はADMINセッション必須**という事実を新たに確認した。

### D-1. blocked-times + BlockedTimeModal → 削除

`admin/blocked-times/page.tsx`と`BlockedTimeModal.tsx`を削除する。ブロック作成機能は`reservations/new`の`isBlocked`トグルとして生きているため機能喪失なし。関連するAPI routeが専用（他から未使用）ならそれも削除対象。

### D-2. カレンダー一本化 → 「正はダッシュボード、admin/calendarはトークン専用に格下げ」

完全削除はしない。`/admin/calendar`はトークン認証トレーナー（セッションなし）の唯一のカレンダー入口を兼ねており、消すとトレーナー導線が壊れる。方針:

- **ADMINの正はダッシュボード**（ナビ「予約」タブ）。月/タイムライン切替は`CalendarView`側（共有コンポーネント）に取り込み、ダッシュボードでも使えるようにする
- `/admin/calendar`は**トークン付きアクセス専用**に縮小。セッション認証のADMINがトークンなしでアクセスしたら`/dashboard?tab=home`へリダイレクト
- 結果、カレンダーの実体は`CalendarView`1つ、入口はセッション用とトークン用の2つ（役割が明確なので重複ではなくなる）

### D-3. 予約作成の入口 → カレンダー起点に1本化、フォームページは維持

`reservations/new`(1266行)のモーダル化はB-3（大規模分割見送り）に反するため**やらない**。ページは残し、導線だけ直す:

- 新規作成ボタンはカレンダー画面（ダッシュボード）に置くのを正とする。一覧(`reservations`)からの導線は残してよいが、それ以外の入口は作らない
- **保存後のリダイレクトを来た場所に戻す**: セッションADMINなら`/dashboard?tab=home`、トークントレーナーなら現行どおり`/admin/calendar?trainerToken=`。これでD-2の「ナビ外画面に飛ばされる」問題が解消する

### D-4. 「その他」グルーピング → 2分類で確定

```
日々の運用      : シフト管理 → オンライン
設定            : トレーナー管理 → 店舗管理 → 配信設定
```

「日々の運用」を上に。セクション見出し付きでカードは現行デザイン踏襲。3分類以上への細分化はしない（5項目に3見出しは過剰）。

### D-5. 「配信設定」への改称 → 今回まとめて実施

`mail-settings`の表示ラベル（その他カード・画面タイトル）を「配信設定」に変更。URLパスの変更は不要（表示名のみ）。

### D-6. 実施順 → Sonnetの提案どおり承認

1. blocked-times+BlockedTimeModal削除 → 2. 改称 → 3. その他2グループ化 → 4. カレンダー統合＋導線修正、の低リスク順で承認。4のみ動作確認必須（トークントレーナーでの表示・予約作成・リダイレクトを必ず確認すること）。

## E. 通知まわりの決定（2026-07-04 / 通知整理メモへの回答）

### E-1. 管理場所 → 案1（配信設定に集約）で確定。分散もハイブリッドも作らない

理由: 通知が届く条件は「pushEnabledフラグ × 端末購読 × 対象者条件」の3段で、状態把握には一覧が必須。入口を増やす（案2/3）と管理者は「どこで切ったか」を探すことになり、シンプル化の方向に反する。案3のショートカットも、実益（配信設定への遷移1回の節約）に対して実装・認知コストが釣り合わないため作らない。**通知のことは配信設定だけ見ればよい**、を管理者のメンタルモデルとして固定する。

### E-2. フラグ粒度 → 1つのまま（現状維持）

1.3の分析を採用。対象者決定ロジック（参加者登録・翌日予約）が実質的なフィルタとして機能しており、種類別フラグは「無関係な人に届かない」問題を二重に解いているだけで実益がない。将来「予約はあるがリマインダーは不要」という個別要望が実際に出た時点で初めて分割を検討する（YAGNI）。その場合も列を増やすのではなく「リマインダーのみ停止」の1オプション追加に留める方針をあらかじめ決めておく。

### E-3. ラベル・説明 → 改善する（低コスト・効果大）

- 会員別タブの見出し「会員ごとのアプリ通知」の直下に説明文1行を追加: **「このスイッチ1つで、予約の確定・変更連絡と前日リマインダー、オンラインレッスン開始前の通知すべてを制御します」**
- トグル列のラベルは「アプリ通知」のままでよい（列名を長くしない。説明は見出し側に持たせる）
- あわせて「ONだが端末未登録」の会員が管理者に分かるように、既存の`pushSubscriptionCount === 0`判定（実装済み）を活かした「端末未登録」バッジ表示があれば維持・なければこの説明文と同時に追加。**「ONなのに届かない」がこの設計最大の混乱源**であり、フラグ粒度よりこちらの可視化が重要

### E-4. 作業範囲と優先順位

コード変更はE-3の説明文・バッジ表示のみ（配信設定画面内で完結、低リスク）。E-1/E-2は「何もしない」が決定内容なので作業なし。D章の残作業（blocked-times削除→改称→2グループ化→カレンダー統合）の**2番（配信設定への改称）と同一PRでE-3を実施**するのが最も効率的。

## F. 管理側「ダイエット詳細」画面の決定（2026-07-04 / diet-planメモへの回答）

前提制約の確認: 会員側の「週間目標」(`WeeklyProgressPanel`)は記録タブ(`WeeklyTab`)＋分析タブ折りたたみに存在しており、**以下の決定はいずれも会員側画面に一切手を触れない**。制約（機能削除・到達困難化の禁止）に抵触しない。

### F-1. 週平均サマリーの統一 → `WeeklyProgressPanel`（バー型）を正とし、カード型2つを削除

- `AdminWeeklySummary`(8カード)と`FoodWeeklySummary`(5カード)は**両方削除**
- 正はバー型。理由: トレーナーの仕事は「実績が目標に対してどうか」の判断であり、実績/目標の対比が構造に組み込まれているのはバー型だけ。カード型は平均値の羅列で判断材料にならない
- ただしカード型だけが持っていた**体重情報が消える**ため、サマリータブ最上部に体重ひとこと（直近値＋先週比。会員側ホームのカード3と同形式）を1行追加する。`AdminWeeklySummary`の`weightDiff`計算はここに流用

### F-2. タブの役割再定義 → 4タブ構成は維持、中身の重複だけ排除

| タブ | 変更後の中身 |
|---|---|
| サマリー | 体重ひとこと ＋ `WeeklyProgressPanel`。**週の全体像はここだけ** |
| 食事（→「食事記録」に改称） | 日別の生データテーブルのみ。週平均カード(`FoodWeeklySummary`)は撤去 |
| 体重・生活 | `AnalyzeTab`のグラフ群。冒頭の折りたたみ週間目標は**管理者文脈では非表示**（F-3） |
| 目標設定 | 現状維持（重複なし・構成妥当） |

タブ数を減らさない理由: 各タブの役割が「全体像/生データ/推移/設定」と直交するようになり、統合する必然がなくなるため。ラベル変更は「食事」→「食事記録」の1箇所のみ（生データ置き場であることを明示）。

### F-3. AnalyzeTab内の折りたたみ週間目標 → propで出し分け

`AnalyzeTab`に`showWeeklyGoals`（デフォルト`true`）を追加し、管理者側diet-planからは`false`を渡す。サマリータブに常設表示があるため管理者文脈では重複。**会員側はデフォルト値により現状のまま変化なし**（制約遵守。会員側の折りたたみを消す判断はしない — 触らないのが最も安全）。

### F-4. 会員側/管理者側の画面統合 → 案2（部分共有）で確定

完全共有（案1）はしない。理由: 両者の問いが違う — 会員は「自分は今日何をすべきか」、トレーナーは「この会員に何を指導すべきか」。案1にすると会員向けのシンプル化（A章の全決定）とトレーナー向けの詳細性が同一コンポーネント内のif分岐として衝突し、双方が劣化する。

共有の線引きを次のとおり固定する:

- **共有する**: データ集計ロジック（`useWeeklyProgress`等のフック）、パネル/ウィジェット単位のコンポーネント（`WeeklyProgressPanel`、`AnalyzeTab`のグラフ群）
- **共有しない**: 画面シェル（タブ構成・レイアウト・何を出すかの選択）

これは既に`ProgressTab`/`AnalyzeTab`で成立しているパターンの明文化であり、「別々に作る意味がない」というオーナーの問題意識には**ロジックと部品の共有**で応える。画面まで統合すると会員側再設計の成果が管理者要件で汚染されるリスクの方が大きい。

### F-5. 着手順 → 重複解消が先、タブ見直しは同時に済む

F-1とF-2は実質同じ作業（コンポーネント2つの削除＋体重ひとこと追加＋ラベル1箇所変更）なので1PRで実施 → F-3（prop追加、小差分）を続けて同PRか直後に。タブ構成の「見直し」は維持決定なので独立作業は発生しない。全体としてD章残作業（カレンダー統合）とは独立しており、並行着手可。

## G. WeeklyProgressPanel 改善の決定（2026-07-04 / 週間目標パネルメモへの回答）

> **注: オーナー仕様の具体化（v2メモ）により、G-1（曜日別達成率ブロック）・G-2（平均への置き換え）・G-5（PR分割）はH章で上書きされた。G-3（体重独立枠・週平均比較の統一）・G-4（3類型）の考え方はH章に引き継がれている。**

共有コンポーネントのため1箇所の変更が会員側（記録・分析）と管理者側（サマリー）の両方に反映される。F-4の共有方針どおりここで直す。

### G-1. 7ブロックの見せ方 → 達成率スケールで統一、絶対値は出さない

- 各ブロックの高さ＝**その日の実績 ÷ 1日目標**（120%でキャップ）。全11項目で統一し、項目別の出し分けはしない。カロリー(kcal)と歩数(歩)ではスケールが4桁違うため絶対値描画は項目間で比較不能になり、達成率なら「どのバーも目標ラインに届いているか」という同じ読み方が全項目に通用する
- **未記録日と実績ゼロを視覚的に区別する**: 未記録＝薄いグレーの空枠、記録あり＝色付きバー（実績0でも極小の色付きを出す）。現行の背景ドット7個はこのブロック列に置き換えて廃止
- ブロック上部に曜日ラベル（月〜日）を極小で付ける。今日のブロックは枠線で示す

### G-2. 数値表示 → 「記録日平均 / 1日目標」を主表示に置き換え、週合計は副次表示

- 主数値: **記録日平均**（例: 「1,480 / 1,650 kcal（4日記録）」）。パーセントもこの比率に変更。オーナー指摘のとおり、累計/週目標は記録日数が少ないと不当に低く見え「記録した日はちゃんとできていたか」が読めない
- 週合計は消さない: 行の下段に小さく「週合計 5,919 kcal」を残す（「週トータルでどれだけ食べたか」の情報喪失を回避。併記だが主従を明確にする）
- **例外: 頻度系（筋トレ回数、既存`isFrequency`）は累計のまま**（「今週 3/4回」）。回数の日平均は無意味。7ブロックは実施日に色が付くだけのシンプル表示にする

### H-2. 1項目あたりの最終レイアウト（4行構成）


- 達成率%は現行計算のまま右肩に維持（2.1）

- 前週比の色は**中立（グレー＋矢印）で統一**。カロリー増＝悪とは限らず、増減の善悪判断はパネルの仕事にしない
- 頻度系（筋トレ）の例外は維持: 行2を「今週 3/4回」、行3は実施日数分色付き、行4なし

## K. 「目標設定」タブ再設計の決定（2026-07-05 / 目標設定メモへの回答）

前提: 「基礎情報からカロリー計算」の初回のみ表示は実装済み・妥当（決定変更なし）。残る4論点への回答。

### K-1. グラフ統合 → 採用。PFC積み上げ1本に統合、カロリー単独グラフは削除

PFC積み上げグラフは各バーの高さ自体が合計kcalであり、カロリー棒グラフは完全に冗長。各バーの上に合計kcal数値ラベルを付け、単独グラフを削除する。

### K-2. 履歴表の廃止＋グラフタップ編集 → 採用。削除はモーダル内に置く

- 統合グラフの各バーをタップ→そのプラン期間の**編集モーダル**を開く（開始日・PFC等を編集可能）。履歴表は廃止
- **削除はこの編集モーダル内の「このプランを削除」ボタン**に置く。グラフ上での直接削除（長押し等）は誤操作リスクが高く作らない
- グラフ直下に「バーをタップすると各プランを編集できます」の1行ヒントを常設（タップ可能であることは見た目から自明でないため）
- 「新しいプランを作成」ボタンは廃止する履歴表からグラフの右上へ移設

### K-3. カロリー入力方式 → 「真実はPFC(g)のみ」に一本化し、カロリー直接書き込みを全廃

矛盾の根源は「カロリーを直接書く経路」と「PFCから導出する経路」の併存。**導出側に一本化**する:

- カロリーは今後もすべての画面で導出値（`P×4 + F×9 + C×4`）・読み取り専用。データとして独立したカロリー値を持たない
- ただしトレーナーの実務は「カロロリーを先に決めてPFCに割る」なので、**「カロリーから配分」導線を新設**: 目標カロリー表示の横にボタンを置き、kcalを入力すると**Pは現値を維持し、残りkcalをF/Cに現在の比率で自動配分**してPFCのg値に変換・反映する（タンパク質は体重基準で決める実務慣行のため固定側に置く）
- 「基礎情報からカロリー計算」の「目標に反映」ボタンも、カロリーを直接書くのをやめて**この同じ変換を通す**。これで「反映した値がP/F/C操作で上書きされて消える」矛盾は構造的に消滅する（もはや消える値が存在しない）

### K-4. 新規プランフォーム → 「現在の目標設定」とフォームを完全統一

簡易版は維持しない。3項目だけの新規作成は糖質/繊維/塩分/生活習慣目標が黙って初期値に落ちる罠であり、意図的仕様だとしても正当化できない。

- フォームコンポーネントを1つに統一し、モード差は**「開始日」フィールドの有無だけ**にする（現在の編集=開始日なし、新規作成=開始日あり）
- 新規作成時は**現在のプランの全値を初期値として複製**（「現在の設定を引き継いで調整する」が実務の自然な流れ。ゼロから6項目＋生活習慣を入れ直させない）
- 導出カロリーの大きな数値表示も統一フォームに含まれるため、「保存するまで何kcalか分からない」問題（3章指摘）も同時に解消
- K-2の編集モーダルもこの統一フォームを流用する（実装が1箇所に集約される）

### K-5. タブの最終構成と実装分割

構成（上から): 初回のみ計算カード → 現在の目標設定（統一フォーム） → 統合グラフ（タップ編集・新規作成ボタン・ヒント行）。履歴表なし。

1. **PR-K1**: グラフ統合＋履歴表廃止＋タップ編集モーダル（既存フォームを仮流用）
2. **PR-K2**: フォーム統一（K-4）＋編集モーダルへの適用
3. **PR-K3**: カロリー入力の一本化（K-3の配分変換、「目標に反映」の変換経由化）

K-3はデータの意味論に触れる変更のため最後に単独で行い、変換ロジック（P固定・F/C比率配分）に単体テストを付ける。

## L. 「目標設定」タブ最終レイアウトの決定（2026-07-05 / 表示過多メモへの回答）

オーナーの3機能整理（現在の設定／新規追加／過去の推移）を設計の憲法として採用し、K章の実装をこれに合わせて減量する。

### L-1. グラフラベル → 常時表示廃止、ホバー/タップのポップアップに変更（採用）

- バー上の常時kcalラベルを削除。ホバー（PC）またはタップ（モバイル）で、**合計kcal＋P/F/Cのg値と比率**をまとめた1つのツールチップを上方向に表示。同時に開くのは1つだけ、グラフ外タップで閉じる
- **K-2のタップ編集との操作競合はこう解消する**: タップ→まずツールチップが開き、その中に「このプランを編集」ボタンを置く→押すと編集モーダル。ワンタップ=情報、ツータップ=編集の2段構え。誤タップで即モーダルが開く事故も防げる（K-2のヒント行文言は「バーをタップすると詳細と編集ボタンを表示」に変更）

### L-2. 「現在の目標設定」→ 閲覧と編集を分離し、コンパクト化

5xl表示は廃止。このタブの仕事は「設定の確認と変更」であり、動機づけ表示（大数字）は場違い。

- デフォルトは**読み取り専用のコンパクトカード**: 「目標 1,650 kcal（P 120g / F 45g / C 190g）」を通常見出しサイズ（2xl程度）で1〜2行＋生活習慣目標を小さな1行（「水分2L・8,000歩・筋トレ週4・睡眠7h」）
- カード右上の「編集」ボタンでK-4の統一フォーム（±ボタン群・カロリーから配分・生活習慣目標を含む）を展開。**±ボタン群が常時見えている現状が「作り込み過剰」の正体**であり、編集モードに畳むことで解消する

### L-3. 3機能で過不足ないか → 過不足なし。ただし2つは「機能の中」に残す

- **基礎情報からカロリー計算**: 削らない。ただし独立セクションではなく「①現在の設定がまだ無いときの空状態表示」と再定義（実装済みの初回のみ表示がまさにこれ。位置づけを明文化するのみ）
- **生活習慣目標の±ボタン群**: 削らない。①の編集モードの中にのみ存在（L-2）。デフォルト画面からは消える
- 削るもの: 5xl数値、常時kcalラベル、常時展開のフォーム。機能の削除はゼロで、**露出の削減だけで3機能整理に適合できる**

### L-4. 最終レイアウト

| 順 | 要素 | 視覚的優先度 |
|---|---|---|
| 1 | 現在の目標設定（コンパクト閲覧カード＋「編集」）※履歴0件時はカロリー計算カードに置換 | 中（情報は簡潔、常に最上部） |
| 2 | 「新しい目標を追加」ボタン（プライマリボタン、カード直下・全幅） | 中〜高（唯一の色付き主ボタン） |

### L-5. 実装分割

1. **PR-L1**: ツールチップ化＋常時ラベル削除＋2段階編集（L-1） — 指摘への即応
2. **PR-L2**: 閲覧/編集分離＋レイアウト再配置（L-2/L-4）

## M. 複数目標の管理・共有の決定（2026-07-05 / 複数目標メモへの回答）

### M-0. 設計の核: 「手段の設定」と「成果のゴール」は別物

カロリー・PFC・生活習慣は期限のない**運用パラメータ（手段の設定）**であり、体重・習慣チャレンジは期限と達成判定を持つ**成果のゴール**。ライフサイクルが違うものを1つのUIに混ぜるのがクラターの原因になる。この区別を土台に、オーナーの「完全に別タブでもよい」という直感を採用する。

### M-1. データモデル → 成果ゴール用の統一テーブルを新設。既存3箇所への後付けはしない

- 新テーブル **`goals`**: `id / user_id / type('weight'|'habit') / title / target_value(体重kgのみ) / start_date / deadline(任意) / status('active'|'achieved'|'missed'|'archived') / achieved_at / note`
- **`diet_goals`は触らない**（期限・達成判定を後付けしない。カロリー設定は「今の運用値＋履歴」という現行モデルが正しい）
- `users.target_weight_kg`は廃止方向: 既存値を各会員の最初の体重ゴール（deadline null）として`goals`に移行し、会員編集画面の入力欄は撤去
- `quit_goals`は「現在日次トラッキング中の習慣リスト」として存続し、**activeな習慣ゴールと同期**させる（ゴール作成→quit_goalsに追加、完了/中止→除去）。日々の○×記録・分析グラフの実装は無変更で生きる

### M-2. タブ構成 → 管理者5タブ化。「目標設定」を「カロリー設定」に改称し、「目標」タブを新設

```
週間パネル / 体重 / グラフ / 目標 / カロリー設定
```

- **カロリー設定**（旧・目標設定）: L章の構成を一切変更しない。名前だけ実態（手段の設定）に合わせる
- **目標**（新設）: L章で実証済みの3機能パターンをそのまま踏襲する —

| 順 | 要素 |
|---|---|
| 1 | 今の目標（activeな体重・習慣ゴールのカード一覧。体重は「8/31までに60kg・あと1.2kg」、習慣は「砂糖をやめる・12日目」） |
| 2 | 「新しい目標を追加」ボタン（種類選択→体重 or 習慣） |
| 3 | 過去の目標（達成✓/未達✗の履歴リスト。グラフにはしない） |

各タブが単機能を保つため、既存タブへの詰め込み（クラター再発）を構造的に回避できる。

### M-3. 体重ゴールのライフサイクル

- **設定・編集・完了操作は「目標」タブ**に一元化。**体重タブは観測専用**: 目標体重ライン（J-3）と「あと◯kg・期限まで◯日」の読み取り表示のみ置き、編集ボタンは置かない（タップで目標タブへ）
- **達成判定は自動検知＋手動確定**: 週平均体重（単日ではなく）が目標値に到達したら、目標タブと週間パネルのハイライト行に「達成条件を満たしました」を表示。トレーナーが「達成にする」を押して確定する（体重は揺れ戻るため自動確定はしない）
- 確定時のフロー: ①達成として履歴へ → ②「次の体重目標を設定」促し → ③「カロリー設定を見直す」導線（カロリー設定タブの計算カードを再表示して開く。K章の初回のみ表示に「達成フローからの明示的再表示」を例外追加）。**カロリーの自動変更はしない**（提案まで。決めるのはトレーナー）
- プッシュ通知は初期スコープ外（毎晩21:00のcronに達成チェックを載せる将来オプションとして記録のみ）

### M-4. 習慣ゴールの完了判定 → 手動完了＋参考情報としてのストリーク

自動判定はしない（「何日続いたら達成か」は目標ごとに恣意的で、ルール化がかえって複雑になる）。目標カードに連続達成日数・達成率を**判断材料として表示**し、完了操作（達成/中止）はトレーナーまたは会員が明示的に行う。期限は任意項目（期限なしの継続チャレンジも許す）。完了した習慣は日次入力の○×リストから消える（M-1の同期）。

### M-5. 「今の目標」共有ビュー → 共通カード1枚を両側に

共通コンポーネント **`CurrentGoalsCard`**: 「カロリー 1,600kcal ／ 体重 8/31までに60kg（あと1.2kg）／ 習慣: 砂糖をやめる（12日目）」の最大3行。

- **管理者側**: 週間パネルタブ最上部（J-5のハイライト行と一体化。「先週比−0.4kg・週目標比92%・記録5/7日」の下に目標行を続ける）
- **会員側**: **プランタブ最上部**に置く。プランタブは元々「何を目指すか」の画面であり意味が一致する。**ホームのカード4枚構成（A-1）は変更しない**（会員ナビも5タブのまま。タブ追加なし＝Q6のナビ影響への回答）
- 会員側の目標タブは作らない。会員による習慣ゴールの完了操作はプランタブのカードから行えるようにする（体重ゴールの確定はトレーナーのみ）

### M-6. 実装分割

1. **PR-M1**: `goals`テーブル＋API＋`target_weight_kg`移行（UI無変更） — 低リスク
2. **PR-M2**: 管理者「目標」タブ新設＋「カロリー設定」改称＋quit_goals同期 → **ここでオーナー実機確認**
3. **PR-M3**: 体重タブ連携（目標ライン・残りkg表示）＋達成の自動検知・手動確定フロー
4. **PR-M4**: `CurrentGoalsCard`（管理者ハイライト統合＋会員プランタブ）

## N. デザインシステム統一の決定（2026-07-06 / デザインシステムメモへの回答）

### N-1. 進め方 → 方針は6軸一括で決定、実装は段階的

色・アイコン・角丸・シャドウ・太さ・フォントは独立変数ではなく、すべて「世界観」（N-4）から導出される従属変数。1軸ずつ決めると軸間の不整合（丸いカードに鋭いアイコン等）が再発するため、**トークン定義は今回一括で確定**し、適用だけを画面単位で段階実施する。

### N-2. デザイントークン＋共通コンポーネント → 導入する。着手はトークン→Button

方針に賛成。順序は「基盤→部品→画面」:

1. **トークン**（Tailwind config＋フォント）が先。部品より先に決めないと部品の実装値が決まらない
2. 部品の第一号は**Button**（211箇所の手書きが最大の揺れ源であり、全画面で使う）。バリアントは primary / secondary / destructive / ghost の4つで打ち止め。続けて Card / Badge / Icon ラッパー
3. 適用画面の先頭は**ダイエットモジュール**（直近の再設計で構造は整理済み。見た目の仕上げを重ねる価値が最も高く、オーナーの目にも一番触れる）

### N-3. アイコン → heroicons採用、頻出10種は一括置換・残りは触った画面から

- ライブラリは**既にdependencyにある`@heroicons/react`（outline系）**を正式採用。lucideへの乗り換えはしない（品質差はなく、依存追加の理由がない）
- 直接importせず**自前の`Icon`ラッパー（`src/components/ui/icons.tsx`）経由**に統一。サイズ・strokeWidth（2固定）をラッパー側で強制し、揺れの再発を仕組みで防ぐ
- 「戻る矢印」「シェブロン」「カレンダー」「閉じる」など**頻出上位10種は一括置換**（機械的にgrepで潰せる）。残りの手書きSVGは画面を触るタイミングで置換
- **UI部品としての絵文字は全廃**（🔴📅⚠️等）。ユーザーが入力した文中の絵文字は対象外

### N-4. 世界観 → 「静かな信頼感」（quiet professionalism）

パーソナルジムの文脈: 会員は体型という繊細な話題を扱っており、騒がしい装飾や過剰なポップさは信頼を削る。派手さではなく「落ち着いていて、正確で、余白がある」を目指す。この一言から各トークンを固定する:

| トークン | 値 | 理由 |
|---|---|---|
| アクセント色 | 1色（前回カラー統一方針の単一アクセント。blue系に集約、indigo/roseは廃止） | 静けさ＝色数の少なさ |
| 角丸 | 3段のみ: カード=`rounded-2xl` / ボタン・入力=`rounded-lg` / バッジ・チップ=`rounded-full` | 柔らかいが緩すぎない |
| シャドウ | 3段のみ: カード=`shadow-sm` / ドロップダウン=`shadow-md` / モーダル=`shadow-xl` | 階層＝浮いている高さ |
| 太さ | 見出し・主数値=`font-semibold`（数値は`tabular-nums`併用） / 本文=`font-normal` / 補足=`text-xs text-gray-500` | ヒエラルキーを太さで作る（1.4の平板さの解消） |
| フォント | **Noto Sans JP**（`next/font/google`、weight 400/600のみ） | 日本語UIに日本語フォント。Interは廃止（1.5の解消） |
| 警告・成功 | semantic色（red/green）は状態表示のみに限定。A-4の「未達に警告色を使わない」原則は全画面に拡張 | — |

### N-5. 優先順位 → デザイン統一が先、販売準備はその後

並行しない。理由: ①「プロが作ったと思われる完成度」は販売デモの前提条件であり、順序が逆だと未完成の見た目で商談することになる ②共通部品化の後にテナント分離を行う方が、分離作業で触る画面がすでに部品化されており手数が減る。デザイン統一（N-6の基準達成）→販売パッケージ化・テナント分離の直列とする。

### N-6. 「プロ品質」の判定基準 → 検証可能な6ゲート＋スクリーンショット審査

主観の「ダサさ」判定を、機械検証できるゲートに置き換える:

1. `ui/Button`外の生`<button>`スタイリング: 対象画面で**0件**（grepで検証）
2. インラインSVGアイコン・絵文字アイコン: **0件**（Icon ラッパー経由のみ）
3. 角丸・シャドウ: トークン3段以外の値が**0件**
4. `font-semibold`が全見出し・主数値に適用され、それ以外の太字が**0件**
5. 日本語グリフを持つフォントが実際に配信されている（DevToolsで確認）
6. アクセント色がblue 1系統のみ（indigo/roseのユーティリティクラス0件）

ベンチマークは**Apple HIGの原則（明瞭さ・階層・一貫性）を判断の参照系**とするが、特定アプリの模倣はしない（他アプリのコピーは世界観の借り物になり、判定も結局主観に戻る）。最終審査は「主要5画面のスクリーンショットを並べ、同一アプリに見えるか」のレビューをオーナーと実施（6ゲート=定量、これ=定性の二重チェック）。

### N-7. 実装分割

## P. パフォーマンス対応の決定（2026-07-07 / パフォーマンス下調べメモへの回答）

### P-1. 優先順位 → 1-1が最優先で正しい

「全期間・無制限のログ取得 × 6コンポーネントが個別再取得」は、データ量が利用期間に比例して増える唯一の問題であり、「使い込むほど遅くなる」というオーナーの体感と機序が一致する。他はすべて定数的な重さ（recharts、select(*)等）で、優先度は下がる。

### P-2. 1-1の対応 → 2段階に分ける。まず日付範囲、次にSWR（ダイエット領域限定）

- **第1段: 日付範囲の絞り込み**（即効・小差分）。`useWeeklyProgress`のfetchに`startDate`/`endDate`を付け、今週＋先週の**2週間分**に限定（前週比計算に必要な範囲。API側は対応済みなので呼び出し側の数行変更のみ）
- **第2段: SWR導入**。ただし**全アプリ書き換えはしない**。`useWeeklyProgress`と体重タブの取得をuseSWR化するだけで、同一キーによる重複排除・キャッシュ共有が効き、「6コンポーネントが個別再取得」問題が消える。ライブラリはSWR（React Queryより軽量で、現状の素朴なfetchからの移行差分が最小）。他画面への展開はN-2と同じ「触った画面から」方式で漸進
- 第1段だけでもデータ量は「全期間→2週間」に落ちるため、第1段リリース後にオーナーの体感を確認してから第2段に進む（第1段で十分ならSWRの緊急度は下がる）

### P-3. 体重タブ → 対象に含める。9週間分に絞る

8週表示＋前週比較の端数を考慮し**9週間**の範囲指定に変更。クライアント側の週別集計ロジックは変更不要（入力が全件→9週分になるだけ）。

### P-4. recharts遅延読み込み → 承認。タブ単位のdynamic importまでやる

4箇所の`next/dynamic`化に加え、**会員メイン画面のタブコンポーネント自体（AnalyzeTab・PlanTab）を`next/dynamic`で遅延読み込み**にする（rechartsだけ剥がすよりタブごと遅延の方が構造が単純で、初回表示のJS量削減も大きい）。ホーム初期表示に必要なのはHomeTab＋共通シェルだけの状態を目指す。ローディング中はスピナーではなく軽いスケルトンを表示。

### P-5. スコープ → 提案どおり承認＋計測を必須条件に

- **今回**: P-2第1段・P-3・P-4（＋触ったAPIルートに限り`select('*')`を必要カラムに置換。全40箇所の一括置換はしない）
- **次回以降**: 管理者ページネーション（現会員数では未発火の問題）、Googleカレンダー直列呼び出し（修正自体は`Promise.allSettled`化で小さいが、部分失敗時のエラー処理仕様を決める必要があり、雑に並列化すると「一部店舗だけ登録漏れ」を握りつぶすリスクがある。仕様を決めてから）
- **計測の義務化**: 着手前後で同一会員のタブ切り替え時の転送量とレスポンスタイムを記録し、改善をオーナーの主観でなく数値で示す（DevTools Network＋`npm run analyze`）。「静的調査による推測」を「計測で裏付けた事実」に変えるのが今回の完了条件
- ブランチは`feature/design-system-n1`に混ぜず**別ブランチ**で（デザインとパフォーマンスは独立してレビュー・リリースできるべき）

## Q. 黒ベース化の決定（2026-07-07 / 黒ベース化メモへの回答）

N-4「白ベース」・O-1「白ベース維持」をオーナー指示により上書きする。1点だけ正直な留意を先に: ジムの明るい照明下や日中の屋外で見る会員（年齢層も広い）にとって、黒ベースは白ベースより可読性の条件が厳しい。以下の決定（純黒回避・コントラスト検証の義務化）はその緩和策込みであり、ダイエットモジュール適用後の実機確認で読みにくさの声が出た場合は微調整の余地を残す。

### Q-1. 理解の確認 → 黒ベースへの「一本化」で正しい

ライト/ダーク切替は作らない（オーナー発言どおり白ベース廃止）。ただしQ-6のトークン化により、将来ライトテーマが欲しくなっても「変数値のセット追加」で済む構造にしておく（切替機能は作らないが、戻れなくなる構造にもしない）。

### Q-2. 黒の階調 → 純黒禁止、zinc系3段のサーフェス

- **base=`zinc-950`（画面地）/ raised=`zinc-900`（カード）/ overlay=`zinc-800`（モーダル・ドロップダウン）**
- 文字: primary=`zinc-100` / secondary=`zinc-400` / muted=`zinc-500`。境界線=`white/10`
- 純黒(#000)は不採用: 有機ELの黒つぶれ・スクロール時の残像・目の疲れに加え、**階層を明度差で作る**（Q-3）ためにはbaseに持ち上げる余地が必要
- 世界観の再言語化: 「静かな信頼感」は維持し、実現技法を差し替える — 白ベースでは「余白と影」で作っていた静けさを、黒ベースでは「**面の明度差と少ない光**（アクセントのオレンジが暗闇で際立つので、使用量はむしろ減らす）」で作る。N-4のシャドウ3段原則は**サーフェス3段原則に置き換え**（影は廃止ではなく`shadow-sm`相当を境界補助として残すのみ）

### Q-3. 適用範囲と順序 → O-7方式を踏襲、ただしトークン化を先行

全画面対象（会員・管理者）。順序は「トークン定義（Q-6）→共通部品（Card/Button/Badge）→ダイエットモジュールで実機確認→会員側残り→管理者側」。O-7のPR列の色部分をこれで上書きし、**未着手だったO-4〜O-6のトークン適用は黒トークン確定後に再開**する（白で塗ってから黒に塗り直す手戻りを防ぐ。これがこの決定を今すぐ確定させる最大の理由）。

### Q-4. 識別色群 → 色相は凍結維持、シェードだけダーク用に再定義

「どの色が何を指すか」の対応表（PFC・予約種別・トレーナー別・曜日）は変えない（再学習コストを発生させない）。変えるのは明度のみ:

- 淡色バッジ: `bg-{color}-100 text-{color}-800` → **`bg-{color}-500/15 text-{color}-300`**（暗い半透明地＋明るい文字）のパターンに機械的に総置換
- 状態色: 黒地でのコントラスト確保のためシェードを上げる（例: 緑=`emerald-400`、赤=`red-400`）。**全状態色・全バッジでWCAG AA（4.5:1）をチェックリスト検証**
- グラフ（recharts）: グリッド線・軸文字を`zinc-700`/`zinc-400`系に、系列色は識別色対応表のダークシェードを使用。食事写真サムネイルは`white/10`の枠線を付けて黒地から浮かせる

### Q-5. 優先順位 → パフォーマンス改善（P章）は並行続行、O-4以降は一時停止

P章は別ブランチかつfetch・ロジック層の変更で、スタイル層とほぼ衝突しないため**止めない**。デザイン側はQ-6のトークン化＋黒転換を最優先し、完了までO-4〜O-6の画面展開を停止（Q-3の理由）。

### Q-6. トークン化 → 先行させる（直書き置換の再演はしない）

色の全面変更はこれで3回目（blue→オレンジ→黒ベース）であり、「直書きを置換して回る」方式がスケールしないことは実証済み。今回を機に**ベース色をセマンティックトークンとしてCSS変数＋Tailwind configに一元化**する:

`surface.base / surface.raised / surface.overlay / text.primary / text.secondary / text.muted / border.subtle / border.strong`

- Card/Buttonなど共通部品はトークン参照に書き換え、直書きの`bg-white`等51ファイルはトークンクラスへ機械置換
- N-6ゲートに**第7ゲート**を追加: 「トークン外のベース色直書き（`bg-white`・`bg-gray-*`・`text-gray-*`等）0件」— 4回目の色変更が来ても変数値の変更だけで済む状態を完成条件とする
- **追記（2026-07-07）**: 淡色ペア`bg-*-50`＋`border-*-100`も白ベース前提の直書きの変種として第7ゲートの検証パターンに含め、残存47ファイルはPR-Q2の機械置換スコープで処理する（個別の先行修正はしない。AnalysisChartCard/AdminStatCardの白浮きバグはこのパターンの実例）

### Q-7. 実装分割

1. **PR-Q1**: セマンティックトークン定義＋Card/Button/Badgeのトークン参照化（この時点では見た目は白のまま＝安全な準備PR）
2. **PR-Q2**: トークン値を黒系に切替＋直書き51ファイルの機械置換＋バッジのダークパターン総置換 → **ダイエットモジュールでオーナー実機確認**
3. **PR-Q3**: グラフ・食事写真・状態色コントラストの個別調整＋AA検証チェックリスト消化
4. 以降: O-4〜O-6の画面展開を黒トークンで再開

## S. 緑色の用途整理の決定（2026-07-07 / 緑色分類メモへの回答）

最上位ルールを1つ立て、そこから4問すべてを導出する: **「緑はstate.success専用。識別色パレットから緑系を除外する」**。緑が「成功」と「脂質」と「通常予約」と「保存ボタン」を同時に意味している状態が緑過多の正体であり、色の意味を1対1に戻す。カテゴリ1の約10箇所は現状維持で正しい。

なおO-3/Q-4の「識別色凍結」はここで一部改訂になるが、Q-4のダークシェード再定義で全色をどうせ触る今が、再学習コストを一度で済ませられる唯一のタイミング。

### S-1. 予約タイプ → 「色は例外にだけ付ける」方式に再設計

虹色スキームを廃止。最も頻度が高い通常予約に緑を塗っていたことが、カレンダー画面の緑過多と「色の意味の希釈」の両方を起こしていた。

| 種別 | 新配色 |
|---|---|
| 通常予約（デフォルト・最頻） | **ニュートラル**（`surface.overlay`＋標準文字。色なし） |
| 体験 | blue |
| ゲスト | purple |
| 予約不可（ブロック） | **赤を廃止**し、muted＋斜線パターン（ブロックは危険ではなく「不在」。赤はstate.danger専用に返却） |
| 月間利用回数バッジ | ニュートラル。**残回数0のときのみ**state.dangerで警告 |

デフォルトが無色になることで、色が付いている=例外・要注意という読み方が成立する。

### S-2. PFC識別色 → F(脂質)のみ変更: emerald→purple

新対応表: **P=amber / F=purple / C=blue**（凍結を1色だけ改訂）。緑をパレットから除外するための最小変更で、P・Cの学習済み対応は温存。purpleはゲスト予約と重なるが、カレンダーと栄養グラフは同居しない別ドメインのため実害なし。この対応表をO-3の規律（全画面で徹底）に載せ直し、再凍結する。

### S-3. 保存済み/下書き等のtransient状態 → 「一時=緑、定常=ニュートラル」ルール

- **一時的な成功フィードバック**（保存直後のトースト・バナー、数秒で消えるもの）= state.success維持（カテゴリ1）

- **CTA・トグル・チェックボックス・選択中ハイライト = brandオレンジに一本化**。緑のCTA・緑トグルは全廃（「第二のアクセント」の廃止。操作系の色は1色という原則をN-4に追記）

- 複数系列 = 識別色パレット（blue / purple / amber / sky / pink。緑・赤は含めない — 系列に使うと状態の意味を勝手に帯びるため）

## T. 商用販売の方針（2026-07-07 / 商用販売メモへの回答）

**免責の明記**: 本章は設計・事業判断の方向づけであり法的助言ではない。利用規約・プライバシーポリシー・契約書はいずれも、雛形作成後に必ず弁護士（個人情報保護法とSaaS契約に明るい人）のレビューを通すこと。以下すべてこの前提の上に立つ。

### T-1. 販売モデル → サブスク（ホスティング継続型）一択で設計する

セルフホスト売り切り(b)は**やらない**。理由: ①Supabase等の継続インフラを前提とした構造で、コード引き渡し後の環境差サポートは初心者にとって最も重い負債になる ②これまでのデザインシステム・トークン化・共通部品化の投資はすべて「1つのコードベースを自分が運用し続ける」前提で価値を持つ ③売り切りは収益が単発で、保守の期待値ズレ（メモ指摘）が構造的に発生する。**テナント分離設計もサブスク前提で確定**する（Q4への回答）。

### T-2. 現実的か → 条件付きでYES。条件は「範囲を絞ること」そのもの

初心者であること自体は障害ではない。障害になるのは範囲の広げすぎ。次の3つの絞りを守る限り現実的:

1. **最初の1社は知人ジムへの有償試験導入**（無償にしない — 無償は「お客」ではなく期待値管理の練習にならない。割引でよい）
2. **ホスティング型サブスクのみ**（コード販売・買い切り・受託カスタマイズの約束をしない）
3. **契約状態・請求は当面手動管理**（Stripe等の課金自動化は2〜3社を超えてから。最初から作り込まない）

### T-3. 絶対にやってはいけないこと（オーナーの問い1・4への回答）

1. **無契約・口約束での導入**。1社目が知人でも、利用規約・免責・サポート範囲（対応時間・保守期間・解約時のデータ返還/削除）を書面にしてから
2. **テナント間のデータ共有・比較**。「他ジムの会員データと比べる」機能は絶対に作らない。体重・食事記録は機密性の高い個人情報であり、テナント境界は事業の信用そのもの
3. **医療的効果をうたう表現**。「痩せる」「改善する」等の断定はアプリ内文言・営業資料の両方で避ける（薬機法・景表法リスク。記録と目標管理のツールであると一貫させる）
4. **第三者規約の未確認展開**。Google Calendar APIとSupabaseの商用・マルチテナント利用条件を**1社目の契約前に**確認する（アカウント停止=全テナント停止、が最大の単一障害点）
5. **削除・返還に応えられない設計**。会員個人・テナント単位のデータ削除とエクスポートが技術的に実行可能であることを、契約前に実装で担保する
6. **個人への責任集中の放置**。2〜3社を超えたら法人化＋賠償責任保険を検討する（最初から法人化は不要）

### T-4. 準備の順序（オーナーへの全体回答）

```
①デザイン統一の完了（進行中: Q/R/S章の消化 = N-5の前提条件）
②法務3点セットの雛形＋弁護士レビュー（規約・プラポリ・免責）※①と並行可
③販売前提のUI/UX実装（T-5。テナント分離より先）
④知人ジム1社に有償試験導入 — 自社+1社の2テナントを手動で運用してみる
⑤④の運用で見えた課題を反映してテナント分離を本実装（T-6）
⑥2〜3社目の営業・法人化検討
```

④を⑤より先に置くのが要点: テナント分離の正しい設計は、実際に2テナント運用して初めて分かる。1社目は「设計のための実地調査」を兼ねる。

### T-5. UI/UX側で先に打つ手（Q2への回答）

- **同意フロー**: 初回ログイン時の利用規約・プライバシーポリシー同意画面（同意日時とバージョンをDB記録。規約改定時の再同意もこの仕組みに乗る）— 販売以前に自社運用でも本来必要なもの
- **常時導線**: 設定画面とログイン画面フッターに規約・プラポリへのリンク
- **データの出口**: 会員単位のデータエクスポート（CSV）と削除依頼の受付導線（T-3-5の実装的裏付け）
- **テナント設定画面の最小形**: ジム名・ロゴ・ブランドカラーの差し替え。**Q-6のセマンティックトークン化がそのままホワイトラベルの技術基盤になる**（変数値の差し替え=テナントテーマ。これまでのデザイン投資が販売機能に直結する）
- 初回オンボーディングフローは1社目導入時の手動サポートで代替し、作り込みは複数社後

### T-6. テナント分離で決定が必要になる項目（洗い出し・Q3への回答）

実装は⑤だが、方向だけ先に固定: **単一DB＋RLS（Row Level Security）による論理分離＋サブドメイン方式**を基本線とする（Supabaseの標準パターンで、テナント数十まではこれで足りる。物理分離=プロジェクト分割は運用コストが初心者向きでない）。⑤時点で決めること: tenant_idの全テーブル導入とRLSポリシー、管理者アカウントのテナント所属、Google Calendar連携のテナント別認証情報、通知(Web Push)のテナント別設定、バックアップとテナント単位リストア手順。

## U. 売り方・価格・プラン設計の方針（2026-07-07 / 商用販売メモ追加相談への回答）

T章の免責を引き継ぐ（価格・営業は事業判断の方向づけであり、最終決定はオーナー自身が1社目の実地で検証する前提）。

### U-1. 売り切り → 選択肢から完全に外す（最終判断）

残さない。決定打はSonnet調査の「逆ざや」: ホスティング構造上、買い切り後もインフラコストと問い合わせ対応は続くのに収入は増えない。「選択肢として残す」こと自体が営業時の迷い（相手に買い切りを求められたら断れるか）になるため、**「月額制のみです」と言い切れる状態にしておくのが初心者にとって最良の防御**。以後の設計・契約書・営業資料はすべてサブスク一本。

### U-2. 価格 → 正規: 月額3万円/店舗を基準線、1社目: 導入特価1.5万円

- **正規価格の基準線: 月額3万円/店舗＋初期導入費5〜10万円**。根拠: 予約管理単体SaaSの相場（1〜5万）に対し、reserve7は「予約＋ダイエット指導管理」の2本柱で単純予約系の上位に位置づけられる。初期導入費は設定代行・データ移行・操作説明という実コストの対価であり、無料にしない（無料にすると導入サポートが「タダの労働」として常態化する）
- **1社目（知人ジム）: 月額1.5万円程度の導入特価**。条件を明文化して交換にする: 期間限定（例: 1年）・フィードバック協力・事例（実名 or 匿名）としての紹介許可。「安くする代わりに何をもらうか」を決めておくのが値引きと投資の分かれ目
- 課金単位は**店舗数**（会員数課金は小規模ジム相手では計算が面倒で敬遠されやすく、店舗数は双方に分かりやすい）
- 迷ったら高い方に倒す: 値下げは後からいつでもできるが、値上げは既存顧客との摩擦になる

### U-3. 機能ティア分け → 今はやらない。1プラン（全機能）＋店舗数課金

松竹梅は**顧客が2〜3社できてから**設計する。理由: ①ティアは「どの画面をどのプランで出すか」の出し分け実装・テスト・営業説明の複雑さを今抱え込むことになり、顧客ゼロの段階では仮説の上に仮説を積むだけ ②reserve7の差別化はダイエット管理であり、それを「上位プラン限定」にすると梅プランは無個性な予約システムになって競合相場に埋没する。

将来ティアを切る場合の候補だけ記録しておく: 梅=予約管理のみ／竹=＋ダイエット管理／松=＋複数店舗・トレーナー管理・分析。実装面の備えは**テナント設定に機能フラグを持てる構造**を意識するのみ（会員ナビには既にプラン別のタブ縮小機構（A-3）があり、この仕組みがそのまま流用できる。今は作らない）。

### U-4. 実際の売り方

- **1社目の見つけ方**: 面識のあるパーソナルジムオーナーが最有力。次点はジムオーナー同士の横の繋がり（器具業者・物件・トレーナー採用のコミュニティ経由）。広告・Web集客はこの段階では不要
- **営業トークの核は「自分のジムで毎日使っている」**。開発話ではなく運用実績として語る: 「予約管理とダイエット指導の記録がこれ1つで回っている。トレーナーは会員を選んで週間パネルを見るだけ」— 機能の列挙ではなく、**トレーナーの指導業務がどう楽になるか**（成果）を売る
- **デモの見せ方**: サンプルデータ入りの**デモテナントを1つ用意**する（自社の実会員データでデモをしない — T-3-2のテナント境界の信用問題に直結）。デモの動線は「会員がスマホで記録する画面（1分）→トレーナーが週間パネルで指導する画面（3分）」の2画面だけに絞る。これはQ/R/S章で磨いた画面そのものであり、デザイン投資の回収点
- **高タッチで入る**: 1社目の初期設定・データ投入・操作説明はオーナー自身が現地/オンラインで手厚くやり、その手順を記録する。これが将来のオンボーディング標準化（T-5で後回しにした部分）の設計図になる

### U-5. UI/UX への影響（この章から発生する実装項目)

新規は2つだけ: ①デモテナント（サンプルデータ生成スクリプト込み。T-6のテナント分離の最初のテストケースを兼ねる） ②テナント設定の機能フラグ構造（U-3。設計だけ先行、UIは作らない）。ティア別の画面出し分け実装は明示的に**着手禁止**とする（U-3が変わるまで）。

## V. 売り切り（シナリオA）の再評価と納品負荷の整理（2026-07-07 / 追加調査への回答）

### V-0. まず訂正: U-1の「逆ざや」論拠は、シナリオAには当てはまらない

オーナーの再質問を受けて整理し直す。U-1で売り切りを退けた主根拠「インフラコストだけ続いて収入が増えない」は、**こちらがホスティングを続ける前提**の話だった。シナリオA（購入側が自分のVercel/Supabaseアカウントを持つ個別納品）ではインフラコストは購入側に移るため、この論拠は消える。オーナーの直感は一部正しい。その上で、残る論点を正確に並べ直したのが本章。**U-1の「完全に外す」は条件付きに改訂する**（V-4）。

### V-1. 「サーバー管理」の切り分けと納品セットアップの工数目安

| 作業 | 内容 | 目安 |
|---|---|---|
| アプリのホスティング | 購入側アカウントでVercelデプロイ＋ドメイン設定 | 30〜60分 |
| DB | Supabaseプロジェクト新規作成＋マイグレーション実行 | 約30分 |
| Googleカレンダー連携 | サービスアカウント方式の6〜7ステップ（審査なし・待ち時間なし） | 30〜60分 |
| 環境変数・通知・動作確認 | env設定、Web PushのVAPIDキー生成、全機能の疎通確認 | 30〜60分 |

**慣れれば1顧客あたり半日（2〜4時間）、最初の1回は予備日込みで1〜2日**を見込む。「Supabase契約だけ」では足りないが、恐れるような作業でもない — ただし非エンジニアのジムオーナーには無理なので、**セットアップ代行は納品価格に必ず含める**（切り売りしない）。

### V-2. 「セットアップ代行込み売り切り」でも残る継続的責任

契約で切れるもの／切れないものを分けるのが要点:

- **契約で管理できる**: 保守期間（例: 納品後30日間の不具合対応のみ、以降は有償スポット）、サポート範囲・対応時間、機能追加は対象外であること。これらを契約書に明記すれば期待値ズレの大半は防げる
- **契約では消えないもの**: ①納品物の重大な欠陥への契約不適合責任（免責特約で制限はできるが完全にゼロにはできない。範囲は弁護士確認事項） ②**Google Calendar API等の外部仕様変更が、納品済みの全コピーを同時に壊すリスク** — 修理義務は契約次第でも、壊れたまま放置すれば評判に跳ね返る ③購入側アカウント起因のトラブル（Supabase無料枠超過・カード期限切れでアプリ停止等）の問い合わせは、責任がなくても事実上こちらに来る
- **構造的な重さ**: 納品コピーは顧客ごとに分岐していく。3社目以降、「どの顧客がどのバージョンか」の管理と修正の横展開がN倍になる。サブスク（1コードベース）との最大の違いはここ

### V-3. チェックリストの着手時期 → 即時。デザイン統一と並行可

- **今すぐ（販売と無関係に必須）**: シークレット類の平文残存の解消・鍵の再生成、実会員データCSVの削除。これは販売準備ではなく**自社のセキュリティ衛生の問題**であり、最優先
- **早期（並行可）**: カレンダーID・`is_admin()`メール等のハードコード解消と環境変数化。env/config層の変更でスタイル層と衝突しないため、Q/R/S章のデザイン作業と完全に並行できる
- どちらもシナリオA/サブスクの両方で必須なので、販売モデルの結論を待つ必要がない

### V-4. 改訂後の結論: 規模で選ぶ

- **1〜2社で終える計画なら、シナリオA（セットアップ代行込み・保守期間限定の売り切り）は成立する**。価格の考え方: 月額基準（U-2の3万円）の24〜36ヶ月分相当＋セットアップ費＝**一括70〜110万円**が出発点。安売りすると「一括で安く買って無限サポート」という最悪の組み合わせになる
- **3社以上に広げる計画なら、サブスク（T/U章）に戻る**。V-2の分岐管理コストが社数に比例して効いてくるため
- どちらでも変わらないもの: V-3のチェックリスト、法務3点セット＋契約書レビュー（T-3）、デモテナント（U-4）。**先に規模の意思を決めるのはオーナー自身**であり、1社目を「シナリオAで納品し、保守契約を月額で別途結ぶ」ハイブリッドから始めて感触を見るのも現実的な折衷案

## W. 販売戦略の統合回答（2026-07-07 / 「他ジムへの販売」検討メモ4章・6.4章への回答）

5章（実運用負荷）はV章で回答済み。ここでは残りの論点に答える。T章の免責（法務・価格は最終的に専門家確認とオーナー判断）を引き継ぐ。

### W-1. シナリオ選択と価格の整合 → Aで開始。ただし10〜20万円の「純粋な売り切り」は構造的に成立しない

V-4のとおり1〜2社はシナリオA。ただしオーナー想定の10〜20万円と、V-4の一括70〜110万円のギャップを正面から言う: **10〜20万円で「保守なし・渡し切り」にすると、セットアップ（半日）＋納品前作業＋最初の問い合わせ対応だけで利益が薄く、以後の無償サポート常態化（V-2）で赤字化する**。10〜20万円の価格帯を活かすなら形を変える:

> **標準オファー: 初期費用 20万円（ライセンス＋セットアップ代行を明示）＋月額保守 1〜2万円（任意だが強く推奨として提示）**

これが「売って終わりに一番近い現実解」（6.4-③への回答）: セットアップ作業は消せない（6.3の調査結論に同意）ので、**消すのではなく価格の中で明示する**。保守を月額オプションに分けることで、買い手は「安く始められる」、売り手は「継続収益と期待値管理」を得る。保守未加入者にはスポット料金表（例: 1件2万円〜）を契約書に添付し、無償対応の常態化を防ぐ。

## X. 需要と二段構え販売の決定（2026-07-07 / 検討メモ7章への回答）

### X-1. 二段構え → 賛成。ただし規律を3つ付ける

「初期数社は買い切り寄り・実績が付いたらサブスク」はV-4（規模で選ぶ）・W-1（初期費用＋月額保守）と整合する正しい設計であり、オーナーのサブスク忌避への答えとしても筋が良い。7.2の分析（未実証ROIへの先払い拒否感・固定費への抵抗は合理的）に同意する。ただし安売りで終わらせないための規律:

1. **社数と期限を先に固定して公表する**（例:「導入事例づくりのため先着3社限定」）。後から通常価格で来た客に説明が立ち、値崩れと「ずっと特価」の期待を防ぐ
2. **交換条件を契約に書く**（U-2と同じ: 事例としての紹介許可・導入効果データの計測協力・フィードバック）。これがないと二段構えは単なる値引き
3. **月額ゼロにするならホスティングは買い手名義（シナリオA納品）に限る**。自社ホスティングのまま月額ゼロは「コストと責任だけ負う」最悪の形なので、自社ホスティングで提供する場合は実費見合い（月5千〜1万円）を必ず取る

勝算の見立て: 第1段（知人圏1〜3社）は高い。第2段が機能するかは**第1段で一次データ（W-3のダッシュボード）を実際に集計するかどうか**にかかっている — これを怠ると二段構えは「安売りしただけ」で終わる。第2段の営業チャネル（紹介が回る仕組み）が最大の不確実性であることは正直に言っておく。

### X-2. 「客単価◯人分」の語り方 → コスト説明には使わない。回収の話に反転させる

「月額＝会員1人分の売上」という表現は、**費用の説明に使うと高さを印象づけるだけ**（相手の頭に「会員を1人失うのと同じ」が残る）。同じ事実を回収側から語る:

> 「**月に1人の退会を防げれば元が取れます**。退会理由の上位は"成果が見えない"ことで、このアプリは会員が自分の進捗を毎日見られるようにするものです」

数字は同じでもアンカーが「出費」から「損益分岐の低さ」に変わる。W-3のROI試算シート（相手の会員数・単価で一緒に計算する）と同じ思想であり、営業トークはすべてこの向きで統一する。

### X-3. 個人事業主・実績なしの立場 → 隠さない。大手SaaSへの対抗軸として使う

隠す戦略は商談で必ず露見して信用を失う。むしろ相手が同じ属性（個人・小規模）だからこそ、これは差別化になる:

- **訴求軸**: 「hacomono等の大手SaaSは複数店舗・大規模ジム向けで、機能も価格も過剰。これは**自分の1〜2店舗のジムを回すために作り、毎日使っているもの**」— 共感軸と実運用証拠の合わせ技。「同じ規模・同じ悩みの経営者が使い込んだ道具」は、大手が構造的に持てない強み
- **前に出すのは運用実績**（「自社で◯ヶ月・会員◯人で運用中」）であって開発力ではない。「自分で作った」を強調しすぎると「この人がいなくなったら終わり」という保守不安を招くため、保守体制の話（W-1の保守契約・スポット料金表）とセットで語る
- 導入実績ゼロは「あなたが最初の導入事例になる。だから特価と手厚いサポートが付く」という第1段オファー（X-1）の根拠として、むしろ明示的に使う

### AB-4[決定]. 文字サイズ4段化の決定（2026-07-20 / AB-3への回答）

角丸と同じく「上限の明文化」で収束させる。以下が確定仕様。

**4段の確定**

| 役割 | クラス | 用途 |
|---|---|---|
| stat（主数値） | `text-3xl font-bold tabular-nums`（＝既存の`.stat-value`） | 体重・カロリー・統計数値など「読ませる数字」すべて |
| heading（見出し） | `text-xl font-semibold` | ページ見出し(h1)もカード/セクション見出しも共通 |
| body（本文） | `text-sm font-normal` | 本文・ラベル・ボタン文字 |
| caption（補足） | `text-xs` ＋ `text-secondary` | キャプション・バッジ・補足 |

`text-6xl`/`text-4xl`/`text-2xl`/`text-lg`/`text-base`は全廃。移行マッピング: 4xl・2xlのページ見出し→heading／4xl・2xlの統計数値→stat／lg→見出しの役割ならheading・本文の強調ならbody+semibold／baseは役割で振り分け（見出し→heading、それ以外→body）。

- **h1のサイズ**: text-2xl/text-4xlの二択には乗らない。**text-xl(heading)に統一**。見出しの格の差は「サイズ」ではなく「位置と余白」で作る（h1=画面最上部、セクション見出し=カード内、同じheadingトークン）
- **主数値**: O-4通りtext-3xl+font-bold+tabular-numsに統一。text-4xl・text-2xlで書かれていた統計数値・数値入力欄の値もすべてstatへ。「数字の大きさ=重要度」の恣意的な使い分けは廃止、重要度は配置と色(text-primary/secondary)で表現する
- **カレンダーセルの極小文字**: 「グリッドセル例外」として明文化して許可。物理的に幅が制約されるカレンダーセル・グラフ軸ラベルのみ、`text-[10px]`を唯一の許可値とする(8pxは読めないため10pxへ引き上げ、11pxは10pxへ寄せる)。散逸防止のため`.text-cell`ユーティリティクラスとして定義し、生の`text-[10px]`直書きは不可
- **⚠️絵文字(text-6xl)**: フォントサイズの問題ではなくN-3「UI部品としての絵文字は全廃」の適用漏れ。heroiconsの警告アイコンに置き換え、サイズはIconラッパーのsize propで管理する(text-6xlは自然消滅)
- **N-6ゲート追加**: 「4トークン(3xl/xl/sm/xs)＋グリッドセル例外(`text-cell`)以外のtext-*使用0件」

## AB. 文字サイズ統一の決定（2026-07-20 / 文字サイズ相談への回答）

角丸と同じく「上限の明文化」で収束させる。以下が確定仕様で、商品診断3.1の表（見出し=text-base案）はこのAB章で上書きする。

### AB-1. 4段の確定

| 役割 | クラス | 用途 |
|---|---|---|
| **stat**（主数値） | `text-3xl font-bold tabular-nums` | 体重・カロリー・統計数値など「読ませる数字」すべて |
| **heading**（見出し） | `text-xl font-semibold` | ページ見出し(h1)もカード/セクション見出しも共通 |
| **body**（本文） | `text-sm font-normal` | 本文・ラベル・ボタン文字 |
| **caption**（補足） | `text-xs` ＋ `text-secondary` | キャプション・バッジ・補足 |

`text-6xl / 4xl / 2xl / lg / base` は**全廃**。移行マッピング: 4xl・2xlのページ見出し→heading／4xl・2xlの統計数値→stat／lg→見出しの役割ならheading・本文の強調ならbody＋semibold／baseは役割で振り分け（見出し→heading、それ以外→body）。

### AB-2. h1は「どちらでもなくtext-xl」

text-2xl vs text-4xlの二択には乗らない。スマホ画面ではtext-4xlのページ見出しは過大で、`text-xl + font-semibold`がiOSのナビゲーションバーやプロのモバイルアプリの標準域。見出しの格の差は「サイズ」ではなく「位置と余白」で作る（h1=画面最上部、セクション見出し=カード内、同じheadingトークン）。サイズが1種類だからこそ迷いと分裂が二度と起きない。

### AB-3. 主数値 → O-4のとおり `text-3xl + font-bold + tabular-nums` に統一

してよい。text-4xl・text-2xlで書かれた統計数値・数値入力欄の値もすべてstatに合わせる。「数字の大きさ=重要度」という恣意的な使い分けを廃止し、重要度は配置と色（text-primary/secondary）で表現する。

### AB-4. カレンダーセルの極小文字 → 名前付き例外として許可（1つだけ・下限あり)

O-2の上限型/下限型と同様に明文化された例外とする: **「グリッドセル例外」— 物理的に幅が制約されるカレンダーセル・グラフ軸ラベルのみ、`text-[10px]`を唯一の許可値とする**。8pxは読めないので禁止（10pxへ引き上げ）、11pxは10px（またはxs）へ寄せて値を1つにする。散逸防止のため`text-cell`のようなユーティリティクラスとして定義し、生の`text-[10px]`直書きは不可。

### AB-5. ⚠️絵文字（text-6xl） → 対象外ではなくIconコンポーネントへ置き換え

これはフォントサイズの問題ではなく**N-3「UI部品としての絵文字は全廃」の適用漏れ**。ExclamationTriangle等のheroiconsに置き換え、サイズはIconラッパーのsize propで管理する（フォントトークンの管轄外）。text-6xlはこれで自然消滅する。

### AB-6. ゲート追加

N-6ゲートに1行追加: 「**4トークン（3xl/xl/sm/xs）＋グリッドセル例外（text-cell）以外の`text-*`使用0件**」。今回13種→5種（4段＋例外1）になり、以後の新画面もこのgrepで守る。
