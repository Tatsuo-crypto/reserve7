# Reserve7 UI整理メモ（Sonnet下調べ・Fable5への申し送り用）

対象: 会員向け画面（1〜5章）＋ 管理者向け画面（6章）。

管理者側は `src/app/dashboard/page.tsx`（下部ナビの中身）と `src/app/admin/*` 配下。

## 1. 現状の画面構成

下部ナビは最大6タブ（プランに`input/analyze/progress`が1つも無効なら3タブに縮小）。

| タブ | コンポーネント | 中身 |
|---|---|---|
| 予約 | ReservationTab | 今後の予約一覧、過去予約の折りたたみ表示 |
| 食事（=DietTab） | InputTab / AnalyzeTab | 「入力」「分析」の2モード切替 |
| ホーム（中央ボタン） | HomeTab | 週間目標の進捗バー（11指標） |
| プラン | PlanTab | 目標カロリー/PFC、目標推移グラフ、目標変更履歴 |
| 通知 | （page.tsx内インライン） | プッシュ通知の許可プロンプト＋説明文のみ |
| オンライン | OnlineTab | 本日のオンラインレッスン枠、Meetリンク |

DietTab内の入力/分析はさらに中身がある。

- **InputTab**（記録入力）: 食事写真OCR解析、現在の体重、水分・歩数・睡眠、筋トレ実施、メモ、その他の目標（禁酒などのquitGoals）
- **AnalyzeTab**（分析）: 期間切替（7日〜すべて）でのカロリー・PFC・塩分・歩数・水分・睡眠などの推移グラフ、体重推移、筋トレカレンダー、習慣達成状況

## 2. 気になった論点

**(a) ホームタブが「数字の羅列」になっている**
HomeTabは週間の実績/目標バーを合計11本（カロリー・P・F・C・糖質・食物繊維・塩分・歩数・水分・睡眠・筋トレ回数）並べているだけで、優先順位がない。しかも肝心の「体重」はここに出てこない（体重推移はAnalyzeTab側だけ）。ユーザーヒアリングでも指摘されていた「数字だらけで疲れる」の実例そのもの。

**(b) ホーム(会員用)とProgressTab(トレーナー用)がほぼ同一ロジックの別実装**
`ProgressTab.tsx`は管理画面`admin/diet-plan`専用で、HomeTabと同じ「週間実績/目標」計算をほぼ丸ごと重複して持っている。UI再設計のタイミングで一本化できると保守コストが下がる（今回のUI議論とは別軸だが、手を入れるなら合わせて検討する価値あり）。

**(c) 「通知」が独立タブなのに中身が薄い**
プッシュ通知の許可プロンプトと説明文だけで単独タブにする必要性が薄い。設定画面やプロフィール配下に格下げできそう。

**(d) オンラインレッスンへの導線が浅い**
レッスン当日はホームで真っ先に見せたい情報のはずだが、現状はホームからもう1タップ必要。

**(e) 予約とホームの役割が近い**
「次の予約はいつか」はホームでもすぐ知りたい情報だが、今のホームには予約情報が一切ない。

## 3. ホームに出す／隠す候補（叩き台）

| 項目 | 出す/隠す候補 | 理由 |
|---|---|---|
| 次回予約（日時・種別） | 出す（最優先） | 「今日/次いつ来るか」は最も聞かれる情報。予約タブへの入口としても機能する |
| 本日のオンラインレッスン（開催時のみ） | 出す（条件付き） | 当日だけ表示すれば良く、常時表示ではない |
| 体重（直近値＋簡易トレンド） | 出す（1指標のみ） | 体重進捗はプロジェクトの主目的機能。ただし数値の棒グラフではなく「先週比」程度の一言サマリーが候補 |
| カロリー実績（本日 or 週間） | 出す（1指標のみ、進捗率のみ） | 主要指標だが、詳細（P/F/C内訳）は分析タブに譲る |
| PFC内訳（P/F/C個別バー） | 隠す | 数字疲れの主因。分析タブに集約 |
| 糖質・食物繊維・塩分の内訳 | 隠す | さらに細かい指標。分析タブのみで十分 |
| 水分・歩数・睡眠 | 隠す（or 1つに集約） | 個別に3本出す必要はなく、「今日やることリスト」的なチェック表現に統合するのも案 |
| 筋トレ実施回数 | 隠す（達成バッジ程度に格上げ） | 数値より「今週◯回達成」のような一言表現向き |
| 通知設定 | ホームから撤去、独立タブも廃止 | プロフィール/設定に統合 |

**設計の方向性（叩き台）**: ホームは「今日/次にやること」に絞ったカード2〜4枚（次回予約・本日のレッスン・体重ひとこと・カロリー進捗ひとこと）にし、細かい数値はすべて「分析」タブへ集約する。

## 4. どの画面から着手するか（優先順位案）

1. **ホーム（HomeTab）** — ユーザーが最初に見る画面であり、今回の課題（数字疲れ）が最も集中している。ここの再設計が全体の見え方を決める。
2. **通知の扱い** — 独立タブを畳むかどうかはナビ構造（6タブ→何タブにするか）に直結するため、ホームの次に決める必要がある。
3. **分析タブ（AnalyzeTab）** — ホームから追い出した詳細指標の受け皿になるため、ホーム再設計と合わせて「何をどこまで見せるか」を整理。
4. **予約・プラン・オンライン** — 現状の情報設計に大きな問題は見当たらないため、ホームとナビ構成が固まった後で微調整すれば良い。

## 5. Fable5に決めてほしいこと（会員側）

- ホームに出すカードの最終選定（上記候補からの取捨選択、表現形式）
- 通知タブを廃止して設定へ統合するか、残すか
- ナビの本数（現行6タブ→減らすか）とラベリング
- 「数字を隠す」際の代替表現（一言サマリー／達成バッジ／グラフ簡略化など）のトーン
- 着手順序の最終承認（上記1〜4の順で良いか）

### 6.3 ナビから孤立している画面（要確認）

以下は`admin/*`配下に実装があるが、上記ナビのどこからもリンクされていない。デッドコード／開発用ページの可能性が高く、残すか消すかの判断が必要。

- `admin/blocked-times/page.tsx` — 予約ブロック時間の管理画面だが、実際の運用は`BlockedTimeModal`（カレンダーから開くモーダル）経由に見える。ページ版が使われているか要確認。
- `admin/calendar-test/page.tsx` — 名称・API呼び出し（`/api/admin/calendar-test`）からして検証用ページ。
- `admin/reservations/page.tsx`・`admin/reservations/new/page.tsx` — `CalendarView`から「予約一覧」的にリンクされているが、カレンダー画面と役割が重複気味（両方から予約を確認・作成できる）。

### 6.4 気になった論点（管理側）

**(a) 「その他」に運営系機能が5つ雑multiに同居**
トレーナー・店舗・シフト・オンライン・通知設定という性質の異なる機能が「その他」という曖昧な1カテゴリに並んでいる。使用頻度（日次運用 vs 月次設定）で分けた方が探しやすい。

**(b) 個別画面の肥大化**
`mail-settings/page.tsx`(1115行)、`diet-plan/page.tsx`(1157行)、`reservations/page.tsx`(666行)、`shifts/page.tsx`(608行)が突出して大きい。1画面に機能を詰め込みすぎている可能性があり、UI再設計時に画面分割も検討候補。

**(c) 予約導線の重複**
カレンダー（ダッシュボード中央タブ）と`admin/reservations`（一覧）の2箇所から予約を見られる。役割分担（カレンダー=日単位の視覚確認、一覧=検索・一括操作 など）を明確にするか、統合するか整理が必要。

**(d) 会員向け「通知」と管理者向け「通知設定」が別物**
会員側の通知タブ（プッシュ許可）と管理者側`mail-settings`（メール・アプリ通知の管理設定）は別画面・別目的だが、名前が近く混同しやすい。ラベリングの整理候補。

### 6.5 Fable5に決めてほしいこと（管理側）

- 孤立している`blocked-times`・`calendar-test`・`reservations`系ページを残す/統合する/削除するかの方針
- 「その他」5項目の再分類（頻度別・カテゴリ別など）とナビ構造の見直し要否
- 肥大化した画面（mail-settings, diet-plan, reservations, shifts）を分割するかどうか
- 会員向け「通知」タブと管理者向け「通知設定」の呼称整理
- 会員側UI再設計（1〜4章）と管理側UI整理のどちらを先に着手するか
