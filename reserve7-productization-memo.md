# Reserve7 「他ジムへの販売」検討メモ(Sonnet下調べ・Fable5への相談)

対象: オーナーから「reserve7を他のパーソナルジムに10万〜20万円程度で売りたい。そのためにセキュリティや譲渡作業、そして『このアプリを入れると売上が今の倍になる』という説得材料が必要では」という相談があった。コードベースを調査した現状を共有し、Fable5に判断・戦略面を相談したい。

---

## 1. セキュリティ・マルチテナンシーの現状(調査結果)

### 1.1 結論: 現状は「1事業者(2店舗)専用」の単一テナント設計

他ジムに"横展開"するには、単なるコピー納品(1社に1セットを渡し切り)なら現状のコードでも動作するが、**同一システムを複数の異なる顧客(ジムオーナー)に量産販売する**には設計変更が必要。

### 1.2 見つかったハードコード(要修正・要ローテーション)

- 管理者判定が完全にハードコード: SQLの`is_admin()`関数(`auth.jwt()->>'email' IN ('tandjgym@gmail.com','tandjgym2goutenn@gmail.com')`)が全RLSポリシーの土台になっている。加えて`src/lib/auth-config.ts`の平文パスワード辞書、その他30箇所以上のAPIルートに同じ2つのメールが直書きされている。
- **RLSポリシーにテナント境界がない**: `USING (auth.uid() = user_id OR is_admin())`という形しかなく、`store_id`等での絞り込みが一切ない。つまり同一Supabaseプロジェクトを複数の異なるジムオーナー顧客で共有すると、管理者ロールの人は**他ジムの会員データも無条件に閲覧できてしまう**。
- ブランディング文字列(「T&J GYM」)がlayout.tsx・manifest.json・README・constants.ts等に散在。
- `.env.local`に本番のSupabase service-roleキー、NEXTAUTH_SECRET(低エントロピー)、Gmailアプリパスワード、Google CalendarサービスアカウントJSON、VAPID鍵が平文で存在(git管理外は確認済みだが、譲渡前提なら全ローテーション必須)。
- **実会員の個人情報を含むCSV/Numbersファイルがリポジトリに含まれている**(`data/会員管理/シート1-在籍.csv`等、実名・メール・料金入り)。これは他社へのコード譲渡以前に、単純なセキュリティ上の是正事項。

## 2. 「アプリを渡す」場合に必要な作業(想定シナリオ別)

### シナリオA: 1社に1セットをそのまま納品する(現状の設計のまま)
- 新規Supabaseプロジェクトを作成し、`supabase/migrations/`配下(現在12本程度)を上から順に流す
- `is_admin()`関数の対象メールをその会社の管理者メールに書き換えて再デプロイ
- `.env`一式をその会社専用に作り直す(全シークレット新規発行、使い回し厳禁)
- ブランディング文字列・ロゴ・manifest.jsonをその会社名に置換
- 実会員データ(会員管理CSV等)がリポジトリに残っていないことを確認してから渡す
- Vercel等のデプロイ先アカウントもその会社名義で新規作成

このシナリオなら**既存コードのままでも技術的には可能**。ただし1社ごとにコードベースが分岐(フォーク)していくため、機能追加やバグ修正を毎回全顧客に手作業で配る運用になる。

### シナリオB: 複数顧客に同一システムをSaaS的に量産販売する
- テナント(顧客ジム)という上位概念を新設し、`store_id`を全テーブル・全RLSポリシーに一貫して効かせる設計に作り替える
- `is_admin()`をハードコードメールから「そのテナントの管理者かどうか」を判定する仕組みに置換
- 顧客ごとのオンボーディング(初期設定・ブランディング・料金プラン設定)を自分で作業しなくて済むようなセルフサーブ導線が理想
- この場合は初期の開発コスト・保守コストが大きく変わるため、10万〜20万円という買い切り価格が成立するかは要検討(シナリオAの"納品"であれば妥当性は考えやすいが、シナリオBの"継続保守が要るSaaS"であれば買い切りより月額課金の方が実態に合う可能性がある)

## 3. 「売上が倍になる」という説得材料について

ここはFable5と相談したい核心。オーナーの意図は「このアプリを導入すると、ジムの売上が数百万円規模で伸びる」という主張だが、これは技術的な調査だけでは断定できない、営業・マーケティングの領域。Sonnetとして提示できるのは以下の材料整理まで。

- **アプリが機能として持っている"売上に効きそうな要素"の棚卸し**: 予約導線の摩擦削減(離脱率低下)、体重・カロリー・PFC・目標管理による継続率向上(退会抑止)、トレーナー側の管理工数削減(1人あたり担当会員数を増やせる=売上上限が上がる)、目標達成の可視化による会員満足度・紹介率向上、など。ただし「これがいくらの売上増につながるか」は**実データ(導入前後の継続率・客単価・トレーナー稼働効率の比較)がないと数字として主張できない**。
- 現状、reserve7自体の運用歴の中で「このアプリ導入前後で継続率がどう変わったか」を示す実データを持っているか(自社の過去データ)が最大の武器になりうる。これがあれば「弊社では導入後に継続率が◯%改善し、月商が◯円伸びた」という一次データの営業トークが作れる。
- 価格設定(10万〜20万円)は買い切りソフトウェアとしては典型的な相場感だが、それを正当化するには「導入後にどれだけの工数削減・売上改善が見込めるか」という定量的なROI試算を先方(見込み顧客)に見せられる状態が望ましい。

### 3.1 オーナーが追加で挙げている「売れる材料」

オーナーからの追加インプット: アプリ単体ではなく、次の要素をセットにした"販売パッケージ"として考えている。

1. **ホームページ(集客用サイト)もそのまま渡せる** — 他プロジェクトで運用しているジムの集客用ホームページを、テンプレートとして他のジムオーナーにもそのまま提供できるのではないか、という案
2. **チラシもテンプレートとして渡せる** — 紙のチラシ素材もセットで提供できるのではないか
3. **ダイエット面の"適切なカロリー設定"がこのアプリの強み** — アラゴン式のカロリー・PFC自動計算機能(K章〜Mプランで実装済み)は、他のパーソナルジムが独自には持っていないであろう差別化要素

この3点セット(アプリ+ホームページ+チラシ+カロリー計算機能)を「導入パッケージ」として売る、という構想について、Fable5の第三者としての意見(勝算があるか、どう訴求するか)を聞きたい、というのが今回のオーナーの狙い。

Sonnet側の技術的な補足: ホームページ・チラシはこのreserve7リポジトリの外(別プロジェクト)にあるとのことなので、それ自体の「他社に渡せる状態か」(ハードコードされた店舗名・電話番号・住所・実写真などが入っていないか)は今回未調査。もしこの2点も販売パッケージに含めるなら、reserve7本体と同様に「固有情報の洗い出しとテンプレート化」が必要になる可能性が高い。

## 4. Fable5に相談したいこと

1. シナリオA(個別納品)とシナリオB(SaaS量産)のどちらを目指す前提で今後の開発を進めるべきか。価格(10万〜20万円の買い切り)は前者に近いが、後者ならサブスク型のほうが実態に合う可能性がある、という論点をどう考えるか
2. マルチテナント化(`is_admin()`のテナント対応化、RLSのstore_idスコープ化)を本気でやる場合、どの程度の開発規模になりそうか、フェーズ分けの提案が欲しい
3. 「売上が倍になる」という売り文句を成立させるために、reserve7自身の運用データ(継続率・客単価等)から何を集め、どう見せるべきか。技術的に「どのデータをダッシュボード化すれば営業資料になるか」の提案
4. セキュリティ是正(ハードコードメール一掃、実データCSVの除去、シークレットローテーション)は販売する・しないに関わらず今すぐ着手すべき項目だと考えるが、優先順位づけを相談したい
5. **「アプリ本体+ホームページテンプレート+チラシテンプレート+アラゴン式カロリー計算」の4点セットを"導入パッケージ"として売るという構想について、第三者視点でどう評価するか。** 特にカロリー計算機能はこのアプリ独自の差別化ポイントとして訴求材料になり得るか、ホームページ・チラシは"テンプレート化してでも他社に渡す価値がある"レベルの資産か、勝率をどう見るか
